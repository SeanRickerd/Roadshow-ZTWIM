== Skill
Learn how to secure database connections using cryptographic workload identity instead of static passwords. You'll deploy PostgreSQL with SPIFFE-issued certificates, establish mutual TLS (mTLS) authentication, and verify that only workloads with valid certificates can connect.

== Objective

* Deploy PostgreSQL server with SPIFFE certificate integration
* Deploy client application with separate SPIFFE identity
* Establish mTLS connection using SPIFFE-issued certificates
* Verify certificate-based authentication (no passwords)
* Inspect certificate properties and identity binding

== Why it Matters
Password-based database authentication creates persistent credential theft targets. Compromised passwords enable lateral movement, and stolen credentials remain valid until manually rotated. Network segmentation alone cannot prevent authenticated attackers from accessing databases. Zero-trust principles require cryptographic proof of workload identity, not just network proximity or static secrets.

== What it Solves

* Credential theft and password spraying attacks
* Lateral movement after initial compromise
* Man-in-the-middle attacks on database connections
* Manual certificate rotation burden
* Identity spoofing via stolen credentials
* Privileged access sprawl from shared database passwords

== Understanding the Attack Surface
[cols="1,2,2",options="header"]
|===
|Area | Risk If Ignored | Analogy
|Static Database Passwords | Credential theft enables persistence | Master key copies floating everywhere
|Password-Based Auth | Brute force / spray attacks | Guessable combination lock
|Network-Only Security | Lateral movement after breach | Locked front door, open internal doors
|Manual Cert Rotation | Stale credentials remain valid | Expired building access badges still work
|Unverified Client Identity | Impersonation via stolen creds | Anyone with password enters as "admin"
|No Mutual Authentication | Server cannot verify client | One-way ID check at secure facility
|Long-Lived Credentials | Extended exposure window | Annual password unchanged
|===

1. *spiffe-helper-init* (init container): Fetches initial certificates before PostgreSQL starts
2. *postgresql-spiffe* (main): PostgreSQL database server configured for SSL with SPIFFE certs

=== Inspect Server Certificate Properties

SPIFFE certificates are automatically generated and mounted at `/opt/postgresql-certs/` by the spiffe-helper sidecar. Let's examine the server's certificate:

[source,sh]
----
oc rsh -n postgresql-spiffe -c postgresql-spiffe deployment/postgresql-spiffe \
  cat /opt/postgresql-certs/svid.pem | openssl x509 -noout -text
----

==== Key Certificate Fields to Examine

*Subject (Certificate Identity)*::
The Common Name (CN) field identifies the server. For PostgreSQL, this should match the server's hostname:
+
[source,text]
----
Subject: C=US, O=SPIRE, CN=postgresql-spiffe.postgresql-spiffe.svc
----

*Issuer (Certificate Authority)*::
Shows the SPIRE server that issued this certificate:
+
[source,text]
----
Issuer: C=US, O=Red Hat, CN=redhat.com
----

*Validity Period (Short-Lived)*::
SPIFFE certificates have intentionally short lifespans (default: 1 hour). This limits the exposure window if a certificate is compromised:
+
[source,text]
----
Validity:
    Not Before: Apr 22 12:32:55 2026 GMT
    Not After : Apr 22 13:32:55 2026 GMT  ← 1-hour validity
----
+
Compare this to traditional certificates that are valid for months or years. Short-lived certificates eliminate the need for certificate revocation lists (CRLs).

*Subject Alternative Name (SAN)*::
Contains the SPIFFE ID URI, which uniquely identifies this workload:
+
[source,text]
----
X509v3 Subject Alternative Name:
    DNS:postgresql-spiffe.postgresql-spiffe.svc
    URI:spiffe://apps.cluster-rp8v4.dynamic.redhatworkshops.io/ns/postgresql-spiffe/sa/postgresql-spiffe
----
+
The SPIFFE ID format: `spiffe://<trust-domain>/ns/<namespace>/sa/<serviceaccount>`
+
This cryptographically binds the certificate to the Kubernetes workload identity (namespace + service account).

*Public Key Algorithm*::
Modern elliptic curve cryptography provides strong security with smaller key sizes:
+
[source,text]
----
Public Key Algorithm: id-ecPublicKey
    Public-Key: (256 bit)
----

==== View Certificate Files

The spiffe-helper maintains three files:

[source,sh]
----
oc rsh -n postgresql-spiffe -c postgresql-spiffe deployment/postgresql-spiffe \
  ls -la /opt/postgresql-certs/
----

Expected output:
[source,text]
----
-rw-------. 1 postgres postgres  241 Apr 22 12:32 svid.key         ← Private key (600 permissions)
-rw-r--r--. 1 postgres postgres 1298 Apr 22 12:32 svid.pem         ← Public certificate
-rw-r--r--. 1 postgres postgres 1403 Apr 22 12:32 svid_bundle.pem  ← CA bundle for verification
----

Key observations:

- *Private key* (`svid.key`): Restricted to owner-only read (600). Never shared across the network.
- *Public certificate* (`svid.pem`): World-readable, sent to clients during TLS handshake
- *CA bundle* (`svid_bundle.pem`): Root and intermediate certificates for verifying other SPIFFE certificates

=== Inspect Client Certificate Properties

The client receives a separate SPIFFE identity. Examine the client certificate:

[source,sh]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \
  cat /opt/postgresql-certs/svid.pem | openssl x509 -noout -text
----

==== Client Certificate Key Fields

*Subject (Client Identity)*::
The CN must match a valid PostgreSQL database user:
+
[source,text]
----
Subject: C=US, O=SPIRE, CN=postgresql_spiffe
----
+
PostgreSQL's `clientcert` authentication mode extracts the CN and maps it directly to a database username. The database initialization script created a user called `postgresql_spiffe`, so this certificate will authenticate as that user.

*SPIFFE ID (Client Workload Identity)*::
[source,text]
----
X509v3 Subject Alternative Name:
    DNS:postgresql_spiffe
    URI:spiffe://apps.cluster-rp8v4.dynamic.redhatworkshops.io/ns/postgresql-spiffe-client/sa/postgresql-spiffe-client
----
+
Note the SPIFFE ID differs from the server:
+
- *Server*: `spiffe://.../ns/postgresql-spiffe/sa/postgresql-spiffe`
- *Client*: `spiffe://.../ns/postgresql-spiffe-client/sa/postgresql-spiffe-client`
+
Each workload gets a unique SPIFFE ID based on its Kubernetes identity (namespace + service account).

*Same Trust Domain*::
Both certificates share the same trust domain (e.g., `apps.cluster-rp8v4.dynamic.redhatworkshops.io`), allowing mutual verification using the shared CA bundle.

=== Establish mTLS Connection from Client

Now we'll connect to PostgreSQL using mutual TLS authentication with SPIFFE certificates.

==== Connect via One-Line Command

You can test the connection directly without entering the pod:

[source,sh]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \

  -c "SELECT current_user, current_database();"
----

Expected output:
[source,text]
----
   current_user    | current_database
-------------------+------------------
 postgresql_spiffe | testdb
(1 row)
----

Success! The connection authenticated using certificates with no password.

==== Interactive Session (Detailed Walkthrough)

For a more thorough exploration, open an interactive shell:

[source,sh]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client
----

Inside the pod, connect to PostgreSQL:

[source,sh]
----
psql "host=postgresql-spiffe.postgresql-spiffe.svc port=5432 user=postgresql_spiffe dbname=testdb sslmode=verify-full sslcert=/opt/postgresql-certs/svid.pem sslkey=/opt/postgresql-certs/svid.key sslrootcert=/opt/postgresql-certs/svid_bundle.pem"
----

==== Understanding the Connection String Parameters

Let's break down each connection parameter:

*host=postgresql-spiffe.postgresql-spiffe.svc*::
Kubernetes service DNS name. Resolves to the PostgreSQL server pod.

*port=5432*::
Standard PostgreSQL port.

*user=postgresql_spiffe*::
Database username. Must match the certificate's Common Name (CN).

*dbname=testdb*::
Database created by the initialization script.

*sslmode=verify-full*::
Strictest SSL mode. Enforces:
+
- Server must present a valid certificate
- Certificate must be signed by a trusted CA (our SPIFFE CA bundle)
- Certificate hostname must match the connection hostname
- Client must present a certificate
+
Other SSL modes (less secure):
+
- `require`: Encrypts but doesn't verify server certificate
- `verify-ca`: Verifies certificate but not hostname
- `prefer`: Uses SSL if available, but falls back to unencrypted

*sslcert=/opt/postgresql-certs/svid.pem*::
Path to client's public certificate (X.509 certificate with SPIFFE ID).

*sslkey=/opt/postgresql-certs/svid.key*::
Path to client's private key. Never transmitted; used for TLS handshake proof-of-possession.

*sslrootcert=/opt/postgresql-certs/svid_bundle.pem*::
CA bundle to verify the server's certificate. Contains SPIRE root and intermediate certificates.

=== Verify Certificate-Based Authentication

Inside the `psql` session, examine the connection details:

[source,sql]
----
-- Show connection information
\conninfo

-- Display SSL connection details
SELECT ssl_is_used();

-- Show database version
SELECT version();

-- Exit
\q
----

Example output:
[source,text]
----
testdb=> \conninfo
You are connected to database "testdb" as user "postgresql_spiffe"
on host "postgresql-spiffe.postgresql-spiffe.svc" at port "5432".
SSL connection (protocol: TLSv1.3, cipher: TLS_AES_256_GCM_SHA384,
bits: 256, compression: off)
----

==== Critical Security Evidence

Analyze what just happened:

1. *No Password Prompt*: The connection succeeded without any password. Authentication used only the client certificate.

2. *User Matches Certificate CN*: The database user `postgresql_spiffe` matches the certificate's Common Name exactly.

3. *TLS 1.3 Encryption*: Latest TLS protocol with forward secrecy.

4. *Strong Cipher Suite*: `TLS_AES_256_GCM_SHA384` provides authenticated encryption.

5. *Mutual Authentication*:
   - Client verified server's certificate (hostname + CA signature)
   - Server verified client's certificate (CN + CA signature)

6. *No Stored Credentials*: No passwords in ConfigMaps, Secrets, or environment variables.

=== Test Security Controls: Connection Without Certificate (Should Fail)

To prove certificate-based authentication is enforced, attempt connection without providing client certificates:

[source,sh]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \
  psql "host=postgresql-spiffe.postgresql-spiffe.svc port=5432 user=postgresql_spiffe dbname=testdb sslmode=require" \
  2>&1 | head -5
----

Expected error:
[source,text]
----
psql: error: connection to server at "postgresql-spiffe.postgresql-spiffe.svc"
(172.231.5.121), port 5432 failed: FATAL: connection requires a valid client certificate
connection to server at "postgresql-spiffe.postgresql-spiffe.svc"
(172.231.5.121), port 5432 failed: FATAL: pg_hba.conf rejects connection for
host "10.x.x.x", user "postgresql_spiffe", database "testdb", no encryption
----

==== Why This Fails

The connection fails for multiple security reasons:

1. *No Client Certificate Provided*
+
We used `sslmode=require` (encryption only) instead of `sslmode=verify-full` (encryption + certificates). No certificate was sent to the server.

2. *PostgreSQL Requires Certificates*
+
The `pg_hba.conf` configuration contains:
+
[source,text]
----
hostssl   all   all   0.0.0.0/0   cert
----
+
This rule requires SSL *and* valid client certificate. No password authentication fallback.

3. *No Password Configured*
+
Even if password auth were allowed, the `postgresql_spiffe` user has no password set. Certificate is the *only* authentication method.

==== What This Proves

- *Zero-Trust Enforcement*: Network connectivity alone is insufficient. Workload must present valid SPIFFE certificate.
- *No Password Bypass*: Cannot authenticate with stolen/guessed passwords because no passwords exist.
- *Defense-in-Depth*: Multiple layers enforcing cryptographic identity (TLS layer + database layer).

=== Observe Automatic Certificate Rotation (Optional)

SPIFFE certificates automatically rotate before expiration with zero downtime. Let's observe this process.

==== Capture Initial Certificate

Record the current certificate's serial number and expiration:

[source,sh]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \
  sh -c 'openssl x509 -in /opt/postgresql-certs/svid.pem -noout -serial -enddate'
----

Example output:
[source,text]
----
serial=1A2B3C4D5E6F7890
notAfter=Apr 22 13:32:54 2026 GMT
----

==== Monitor spiffe-helper Logs

The spiffe-helper sidecar logs certificate rotation events:

[source,sh]
----
oc logs -n postgresql-spiffe-client -l app=postgresql-spiffe-client -c spiffe-helper --tail=20 -f
----

You'll see entries like:
[source,text]
----
time="2026-04-22T13:02:54Z" level=info msg="Received update" spiffe_id="spiffe://..." system=spiffe-helper
time="2026-04-22T13:02:54Z" level=info msg="X.509 certificates updated" system=spiffe-helper
----

The helper fetches new certificates approximately 30 minutes before expiration (configurable).

==== Verify New Certificate After Rotation

After observing a rotation event in the logs, check the certificate again:

[source,sh]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \
  sh -c 'openssl x509 -in /opt/postgresql-certs/svid.pem -noout -serial -enddate'
----

The serial number will be different, and the expiration time will be extended:
[source,text]
----
serial=9F8E7D6C5B4A3210  ← New serial number
notAfter=Apr 22 14:32:54 2026 GMT  ← New expiration (1 hour from rotation)
----

==== Test Connection After Rotation

Verify database connections continue working with the new certificate:

[source,sh]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \
  psql "host=postgresql-spiffe.postgresql-spiffe.svc port=5432 user=postgresql_spiffe dbname=testdb sslmode=verify-full sslcert=/opt/postgresql-certs/svid.pem sslkey=/opt/postgresql-certs/svid.key sslrootcert=/opt/postgresql-certs/svid_bundle.pem" \
  -c "SELECT 'Connection successful after rotation' AS status;"
----

Expected output:
[source,text]
----
             status
-----------------------------------
 Connection successful after rotation
(1 row)
----

==== Key Observations

- *Zero Downtime*: Database connections continue during rotation
- *Automatic Process*: No manual intervention required
- *Short Validity Windows*: Each certificate is only valid for ~1 hour
- *Continuous Renewal*: New certificates fetched before old ones expire
- *No Certificate Revocation Lists (CRLs) Needed*: Short TTL makes revocation unnecessary

=== Troubleshooting

==== Pods Not Starting

*Symptom*: Pods stuck in `Pending`, `ImagePullBackOff`, or `CreateContainerConfigError`

*Check*:
[source,sh]
----
# View pod status
oc get pods -n postgresql-spiffe

# Check events
oc get events -n postgresql-spiffe --sort-by='.lastTimestamp' | tail -20

# Describe pod for detailed error
oc describe pod -n postgresql-spiffe <pod-name>
----

*Common Causes*:
- *ImagePullBackOff*: Image registry unreachable or requires authentication
- *CreateContainerConfigError*: CSI driver not running or volume mount failed
- *CrashLoopBackOff*: Container starting but failing - check logs

==== CSI Volume Mount Failures

*Symptom*: Error like "unable to attach or mount volumes"

*Check*:
[source,sh]
----
# Verify CSI driver is running
oc get pods -n openshift-zero-trust-workload-identity-manager -l app.kubernetes.io/name=spire-spiffe-csi-driver

# Verify CSI driver registration
oc get csidriver csi.spiffe.io

# Check SPIRE agents are running
oc get pods -n openshift-zero-trust-workload-identity-manager -l app.kubernetes.io/name=spire-agent
----

*Solution*:
- Ensure SPIRE agents running on all nodes (one pod per node)
- Verify CSI driver DaemonSet healthy
- Check CSI driver logs: `oc logs -n openshift-zero-trust-workload-identity-manager <csi-driver-pod> -c spiffe-csi-driver`

==== Certificate Not Generated

*Symptom*: Directory `/opt/postgresql-certs/` empty or missing files

*Check*:
[source,sh]
----
# Check if ClusterSPIFFEID exists
oc get clusterspiffeid -A

# Check spiffe-helper init container logs
oc logs -n postgresql-spiffe <pod-name> -c spiffe-helper-init

# Check spiffe-helper sidecar logs
oc logs -n postgresql-spiffe <pod-name> -c spiffe-helper
----

*Common Causes*:
- ClusterSPIFFEID resource not created or misconfigured
- SPIRE agent cannot communicate with SPIRE server
- Workload attestation failed (namespace/SA selectors don't match)

==== Connection Failures

*Symptom*: `psql` command times out or shows connection refused

*Check*:
[source,sh]
----
# Verify PostgreSQL service exists
oc get svc -n postgresql-spiffe

# Check PostgreSQL pod is running
oc get pods -n postgresql-spiffe -l app=postgresql-spiffe

# Check PostgreSQL logs
oc logs -n postgresql-spiffe <pod-name> -c postgresql-spiffe

# Test network connectivity
oc rsh -n postgresql-spiffe-client <client-pod> -c postgresql-spiffe-client \
  curl -v telnet://postgresql-spiffe.postgresql-spiffe.svc:5432
----

*Common Causes*:
- PostgreSQL pod not running or not ready
- Service selector doesn't match pod labels
- Network policies blocking traffic (if NetworkPolicies are configured)

==== SSL/Certificate Errors

*Symptom*: "SSL connection has been closed unexpectedly" or "certificate verify failed"

*Check*:
[source,sh]
----
# Verify certificates exist
oc rsh -n postgresql-spiffe-client <client-pod> -c postgresql-spiffe-client \
  ls -la /opt/postgresql-certs/

# Check certificate validity
oc rsh -n postgresql-spiffe-client <client-pod> -c postgresql-spiffe-client \
  openssl x509 -in /opt/postgresql-certs/svid.pem -noout -dates

# Verify certificate CN
oc rsh -n postgresql-spiffe-client <client-pod> -c postgresql-spiffe-client \
  openssl x509 -in /opt/postgresql-certs/svid.pem -noout -subject
----

*Common Causes*:
- Certificate expired (check notAfter date vs current time)
- Certificate CN doesn't match database username
- Certificate not signed by trusted CA (trust domain mismatch)
- Wrong sslmode in connection string

==== Permission Denied Errors

*Symptom*: "permission denied" when accessing certificate files

*Check*:
[source,sh]
----
# Check file permissions
oc rsh -n postgresql-spiffe <pod-name> -c postgresql-spiffe \
  ls -la /opt/postgresql-certs/

# Check container user
oc rsh -n postgresql-spiffe <pod-name> -c postgresql-spiffe id
----

*Solution*:
- Ensure `anyuid` SCC is granted to ServiceAccount
- Verify `runAsUser: 26` and `fsGroup: 26` in pod securityContext
- Check spiffe-helper is running as UID 26 (postgres user)

== FAQs
Why not just use strong passwords?:: Passwords are static targets; certificates rotate automatically and cannot be brute-forced.
What if the SPIRE server is compromised?:: Certificate rotation limits exposure window; compromised SPIRE requires cluster-level breach, not just application compromise.
Can I still use password auth for admin access?:: Yes, configure `pg_hba.conf` with different rules for different users; certificate auth for applications, password for break-glass admin.
How often do certificates rotate?:: Default is typically 1 hour; configurable via ClusterSPIFFEID TTL settings.
What happens during certificate rotation?:: spiffe-helper automatically fetches new certificates before expiration and signals PostgreSQL to reload; no connection interruption.
Why is CN set to the database username?:: PostgreSQL's certificate authentication maps certificate CN to database user; this enforces identity binding at the TLS layer.

== Closing Story
Certificate-based authentication is like airport security with biometric ID: you cannot borrow someone else's fingerprint, and the ID expires if you leave and return. Passwords are like shared gate codes: anyone with the code enters, and the code persists until manually changed.
